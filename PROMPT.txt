我们机构搞 K12 课后辅导，每个季度末有一次综合大考，考完要出一份详细的评分报告。原来是老师手动批改主观题（简答、论述、材料分析），300 多份卷子要批一个礼拜。这次想用 Go 写个自动评分系统，主观题部分让系统先打一个初评，老师再在初评基础上微调，最后定稿。

客观题部分全自动。单选题选对满分、选错 0 分；多选题按选项匹配打分，全对得满分、漏选按正确选项数比例给分（正确答案 ABC 学生选 AB 没选 C，得三分之二）、错选或者多选了别的直接 0 分；填空题精确匹配（abacus、photosynthesis 这种专有名词不允许同义替换），答案里可能有多个空，每个空独立判分；判断题选对得满分选错 0 分。

主观题打分逻辑要细做。简答题是给关键词的方式——出题老师在题目里配一个关键词列表（5 到 10 个），学生答对几个关键词得对应比例的分。比如满分 10 分、5 个关键词，答对 3 个得 6 分。论述题和材料分析题复杂一些——同时看关键词覆盖率、答案长度、跟参考答案的相似度（用字符级公共子序列长度除以参考答案长度算相似度比），三个指标加权：关键词 50%、长度 20%、相似度 30%。长度过短（比如 50 字以下）直接扣掉 30% 分值防止学生蒙。

评分流程是三段式。初评是系统跑出来的分数，自动进答卷；复核是另一位老师在线修改（可以改分数也可以写评语）；终评是学科组长签字定稿。三个阶段在答卷上有不同状态标识，老师登录后看到的是他该看到的阶段——初评老师只看初评队列、复核老师只看复核队列。同一张答卷不能同时被两个人改，加锁机制要做。

分数段位划分：90 分以上优秀、75 到 89 良好、60 到 74 合格、60 分以下不合格。这个段位在评分定稿后自动计算，写到学生成绩单里。

成绩单要能导出成 PDF 和 Excel。PDF 给学生看有学校抬头和分数段位、雷达图（学业表现那几门课）、教师评语；Excel 给教务处存档，结构化数据为主。导出任务量可能一次几百份，要支持异步导出，先生成一个任务 ID，导出完成再下载。

数据存内存就行，启动时从一个 JSON 配置文件读入题库和学生名单，进程内的修改用 map 加锁保护。重启数据丢一些能接受，定稿的试卷另外存一份 JSON 快照。

Go 这边用 gin 框架，分层做清楚——handler 接收请求、service 算分数、repository 管数据、model 定义几个结构体（试卷、题目、学生、答卷、评分记录）。关键算法单独抽出来放一个 package 方便以后调整。

最后有个细节，从 URL 里传入一个试卷 ID 加学生 ID 能直接跳到该学生该试卷的答卷详情页，老师从群里点链接就能开始批改。

服务跑起来访问首页应该看到试卷列表，点进具体试卷看到所有学生的初评进度条和待批改队列。
